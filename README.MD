# SkillBridge API: Educational Management & Monitoring System

SkillBridge is a vocational training management backend built with FastAPI. It features a sophisticated dual-layer security model and a relational database schema designed to track multi-institution educational progress.

## 1. Live API & Deployment Notes

- **Base URL:** `http://localhost:8000` (local development)
- **Interactive Docs:** `http://localhost:8000/docs` (Swagger UI)

**Deployment Note:**  
This API is configured for local production environments using Uvicorn. For cloud deployment (for example AWS or Heroku), ensure `SECRET_KEY` and `MONITORING_API_KEY` are set through environment variables rather than `.env` files.

4.2. Trainer & Batch Flow

4.2.A. Signup & Login (Institution & Trainer)

To create a batch, we first need an Institution, and then a Trainer linked to that Institution.

```powershell
# 1. Signup Institution
$inst = Invoke-RestMethod -Method POST -Uri "$baseUrl/auth/signup" `
    -Headers @{"Content-Type"="application/json"} `
    -Body '{"name":"Alpha Academy", "email":"admin_final@alpha.com", "password":"password", "role":"institution"}'

# 2. Signup Trainer (Linked to the Institution ID created above)
Invoke-RestMethod -Method POST -Uri "$baseUrl/auth/signup" `
    -Headers @{"Content-Type"="application/json"} `
    -Body "{`"name`":`"Trainer Joe`", `"email`":`"trainer_final@test.com`", `"password`":`"password`", `"role`":`"trainer`", `"institution_id`": $($inst.id)}"

# 3. Login Trainer
$login = Invoke-RestMethod -Method POST -Uri "$baseUrl/auth/login" `
    -Headers @{"Content-Type"="application/json"} `
    -Body '{"email":"trainer_final@test.com","password":"password"}'
```

4.2.B. Create Batch, Invite Token & Session

A Trainer must generate an invite token so students can join the batch.

```powershell
# 1. Create Batch
$batch = Invoke-RestMethod -Method POST -Uri "$baseUrl/batches" `
    -Headers @{"Authorization"="Bearer $($login.access_token)"; "Content-Type"="application/json"} `
    -Body '{"name":"Cloud Computing 101"}'

# 2. Generate Invite Token (CRITICAL: Required for students to join)
$invite = Invoke-RestMethod -Method POST -Uri "$baseUrl/batches/$($batch.batch_id)/invite" `
    -Headers @{"Authorization"="Bearer $($login.access_token)"}

# 3. Create Session
$session = Invoke-RestMethod -Method POST -Uri "$baseUrl/sessions" `
    -Headers @{"Authorization"="Bearer $($login.access_token)"; "Content-Type"="application/json"} `
    -Body "{`"title`":`"FastAPI Setup`", `"date`":`"2024-12-01T00:00:00`", `"start_time`":`"09:00:00`", `"end_time`":`"11:00:00`", `"batch_id`": $($batch.batch_id)}"
```

4.3. Student Flow

4.3.A. Signup & Login (Student)

```powershell
# 1. Signup Student
Invoke-RestMethod -Method POST -Uri "$baseUrl/auth/signup" `
    -Headers @{"Content-Type"="application/json"} `
    -Body '{"name":"Student Sam", "email":"sam_final@test.com", "password":"password", "role":"student"}'

# 2. Login Student
$sLogin = Invoke-RestMethod -Method POST -Uri "$baseUrl/auth/login" `
    -Headers @{"Content-Type"="application/json"} `
    -Body '{"email":"sam_final@test.com","password":"password"}'
```

4.3.B. Join Batch & Mark Attendance

The student must join the batch using the invite token before marking attendance.

```powershell
# 1. Join Batch (Uses the token generated by the Trainer in Step 2B)
Invoke-RestMethod -Method POST -Uri "$baseUrl/batches/join" `
    -Headers @{"Authorization"="Bearer $($sLogin.access_token)"; "Content-Type"="application/json"} `
    -Body "{`"token`": `"$($invite.invite_token)`"}"

# 2. Mark Attendance (This will now succeed with a 200 OK)
Invoke-RestMethod -Method POST -Uri "$baseUrl/attendance/mark" `
    -Headers @{"Authorization"="Bearer $($sLogin.access_token)"; "Content-Type"="application/json"} `
    -Body "{`"session_id`": $($session.session_id), `"status`": `"present`"}"
```

4.4. Monitoring Officer Flow

4.4.A. Login & Token Exchange

```powershell
# 1. Initial Login
Invoke-RestMethod -Method POST -Uri "$baseUrl/auth/signup" `
    -Headers @{"Content-Type"="application/json"} `
    -Body '{"name":"Officer Smith", "email":"officer_final@gov.com", "password":"password", "role":"monitoring_officer"}'

$moLogin = Invoke-RestMethod -Method POST -Uri "$baseUrl/auth/login" `
    -Headers @{"Content-Type"="application/json"} `
    -Body '{"email":"officer_final@gov.com","password":"password"}'

# 2. Get Scoped Monitoring Token (1-hour expiry)
$scoped = Invoke-RestMethod -Method POST -Uri "$baseUrl/auth/monitoring-token" `
    -Headers @{"Authorization"="Bearer $($moLogin.access_token)"; "Content-Type"="application/json"} `
    -Body '{"key":"SKILLBRIDGE_ADMIN_2024"}'
```

4.4.B. Fetch Protected Data

```powershell
Invoke-RestMethod -Method GET -Uri "$baseUrl/monitoring/attendance" `
    -Headers @{"Authorization"="Bearer $($scoped.monitoring_access_token)"}
```

## 8. Schema & Design Decisions

### Many-to-Many (`batch_trainers`)
Vocational training often requires multiple subject matter experts for one batch. Using an association table allows for flexible trainer assignments without duplicating batch data.

### `batch_invites` (Invite Tokens)
To prevent unauthorized enrollment, students cannot join a batch by ID. They must possess a cryptographically secure token generated by a trainer, which is invalidated upon use.

### Dual-Token Approach
To satisfy Task 2’s high-security requirement, Monitoring Officer access is decoupled from standard login. Even if an officer’s password is compromised, the `/monitoring/attendance` endpoint remains locked unless the secondary system API key is provided to generate a short-lived scoped token.

## 9. Implementation Status

### Fully Working
- JWT authentication and bcrypt password hashing.
- RBAC for all 5 roles: Institution, Trainer, Student, Officer, Manager.
- Dual-token exchange flow for monitoring data.
- Automated database seeding with relational integrity.
- Batch creation and student-to-batch mapping.

### Partially Done
- Student enrollment flow: the backend logic for invite validation is present, but the automated join endpoint is simulated via the seed script.

### Skipped
- Frontend interface: as per requirements, the focus remained on the robust CLI-runnable API.
- File uploads: attendance is tracked via JSON data rather than image or biometric uploads.

## 10. Future Improvements

If given more time, I would implement token blacklisting. Currently, JWTs are stateless and valid until expiration. Implementing a Redis-backed blacklist would allow for immediate logout capabilities and enhanced security if a Monitoring Officer’s device is reported lost or compromised.
